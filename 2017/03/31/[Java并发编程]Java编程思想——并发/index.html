Java编程思想——并发

发表于 2017-03-31   |   分类于 Java并发编程   |  

思维导图

线程与进程

线程资源同步和交互机制

重点是：JVM中线程资源同步机制及线程之间交互的机制

线程资源同步

i++操作实质：

i++动作在JVM中分为装载i、读取i、进行i+1操作、存储i、及写入i五个步骤

线程之间的共享变量存储在主内存（main memory）中，每个线程都有一个私有的本地内存（local memory），本地内存中存储了该线程以读/写共享变量的副本。

对于一个简单的 i++ 操作，结合上图1 ，会发生如下的步骤

  • read：作用于主内存中，把主内存中一个变量的值传输到 工作内存 中。
  • load：作用于工作内存，把从read 操作从主内存中得到的值放入到工作内存的副本中。
  • use：把工作内存中的该副本值传递给执行引擎（也就是操作数栈中）。
  • assign：作用于工作内存，把执行引擎执行后的新值传递给该工作内存的变量。
  • store：作用于工作内存，把工作内存中该变量的值传送到 主内存 中去。
  • write：作用于主内存的变量，把store 操作 得到的值写入到 主内存的该变量中。

前三步（read、load、use）完成变量值的读取，在虚拟机栈内完成i+1操作，后三步将变量写入主内存

所以说，一个 i++ 操作并不是原子性（原子性：一些程序语句，必须连续执行，中间不能被打断）的。这上述的这些步骤中，可能会有其他线程对主内存的变量进行操作，从而导致出现多线程问题。

同步机制

  • synchronized
  • volatile
  • lock/unlock机制

1、synchronized

public synchronized int getNextId(){
    return i++;
}

线程A执行getNextId方法时，在对象实例ID加上一个lock，然后继续执行return i++；此时如果线程B并发访问getNextId()方法时，发现这个对象实例ID有lock，于是将线程B放入等待队列，只有当线程A执行完成后，释放对象实例ID上的lock，当下次线程调度线程B时，线程B才可以执行

synchronized方法锁的对象是当前对象实例，当synchronized标记在一个static方法上，那么锁放在当前Class实例上

2、lock/unlock

使用lock/unlock编写多线程程序时，一定要保证lock和unlock是成对出现的，并且保证lock后程序完毕后，一定要unlock

3、volatile

控制对象的可见性，并不能保证对象操作的原子性；定义volatile的对象，线程不会将其从主内存复制到本地内存中，而是直接在主内存中进行操作

适用于多线程读操作，写操作需要加锁

线程交互机制

线程之间除了产生资源竞争外，还会有交互的需求。比如连接池中会有get和return操作，当get方法时可用的线程数为0时，需要进入一个等待状态，当有连接返回时，应该通知get方法，不需要再等待了。如果没有交互机制，只能在get中不断轮训判断可使用的连接数了

JVM提供wait/notify/notifyAll方法支持这类需求

案例

public class ConnectionPool {

    //两个连接
    private int[] connections = new int[]{1, 2};
    private int current = 0;

    public synchronized int get() throws InterruptedException {
        while (current == connections.length) {
            System.out.println(Thread.currentThread().getName() + "在等待");
            this.wait();
        }

        System.out.println(Thread.currentThread().getName() + "获得连接");
        return connections[current++];

    }

    public synchronized void release() {
        if (current > 0) {
            current--;
        }
        System.out.println(Thread.currentThread().getName() + "释放连接");
        notify();
    }

    public static void main(String[] args) throws InterruptedException {
        final ConnectionPool connectionPool = new ConnectionPool();
        new Thread(new Runnable() {
            public void run() {
                try {
                    connectionPool.get();
                    Thread.sleep(2000);
                    connectionPool.release();
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        }).start();


        new Thread(new Runnable() {
            public void run() {
                try {
                    connectionPool.get();
                    Thread.sleep(2000);
                    connectionPool.release();
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        }).start();

        new Thread(new Runnable() {
            public void run() {
                try {
                    connectionPool.get();
                    Thread.sleep(2000);
                    connectionPool.release();
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        }).start();
    }
}

结果

Thread-0获得连接
Thread-1获得连接
Thread-2在等待
Thread-0释放连接
Thread-2获得连接
Thread-1释放连接
Thread-2释放连接

调用Object的wait方法可以让当前线程进入等待状态，程序不能往下执行，只有调用了此Object的notify、或notifyAll方法，或者wait到达了指定的时间后，才会激活继续执行

notify只是随机找处于等待此Object的一个线程；notifyAll是通知wait此Object的所有线程

为了放置假唤醒，提高程序的可靠性，wait被唤醒后，需要再次判断等待的状态是否被更改了，如果未更改，则继续进入wait状态，这种做法称作双重检查。查看一下代码，即加上了while进行双重检查，而且wait操作写在最前面

wait/notify机制原理：

当调用了对象的wait方法后，JVM程序执行引擎将此线程放入一个wait集合中，并释放此对象的锁（本来加上了synchronized只能有一个线程处于等待状态，现在可以多个线程处于等待状态）

  • 当其他线程调用此对象的notify方法时，会从wait集合中中随机找一个等待的此对象上的线程，并将其从wait集合中删除，JVM线程执行引擎就可以再次调度此线程了

  • 当使用notifyAll方法时，则会删除wait集合中所有等待此对象的线程

线程状态及分析方法

  • 线程创建完毕后进入new状态
  • 调用了线程的start方法后，线程进入Runnable状态，放入jVM的可运行线程队列中，等待CPU的执行权
  • 当线程执行时，线程状态变为Running
  • 线程执行sleep、wait、join或者进入了IO阻塞、锁等待时，则进入Wait或Block状态
